Phân Tích Kiến Trúc

3. Giải Thích Chi Tiết Cơ Chế Hoạt Động

Mô Hình Wire Provider (TLS ➔ AWS)

Đây là kỹ thuật liên kết providers (Wire Providers) đặc trưng trong Terraform.

Tài nguyên tls_private_key sinh cặp khóa hoàn toàn bằng phần mềm, sau đó truyền Public Key trực tiếp sang aws_key_pair làm đầu vào.

Đồng thời, local_file lưu Private Key dạng file .pem xuống local máy dev để SSH trực tiếp vào EC2 mà không cần tạo Key Pair thủ công trên AWS Console.

Kind Cluster & NodePort Mapping

Do Kind chạy Cluster bằng các Docker Container đóng vai trò là Kubernetes Nodes, dải NodePort mặc định của Kubernetes (cụ thể là port 30080) nằm gọn bên trong container điều khiển (control-plane).

Thông qua tệp cấu hình kind-config.yaml được tạo ở user_data.sh, cấu hình extraPortMappings sẽ map cổng 30080 của container Control-Plane ra cổng 30080 của máy chủ EC2. Nhờ đó, ALB Target Group mới có thể kết nối từ ngoài.

Kubernetes API Proxy & Bảo Mật

Để Terraform tại máy local của bạn hoặc runner có thể deploy các tài nguyên K8s (Deployment, Service) trực tiếp, chúng ta cần truy cập được vào API Server của Cluster.

Ở bước cuối của user_data.sh, lệnh kubectl proxy --port=8081 --address=0.0.0.0 được chạy ngầm. Cổng 8081 này đóng vai trò cầu nối, chỉ nhận request từ Security Group được chỉ định để bảo mật tối đa cho API Server.

Quản Lý Providers trong Terraform

Terraform quản lý vòng đời tài nguyên thông qua 4 Providers chính khai báo trong providers.tf:

1. AWS Provider: Đóng vai trò dịch mã HCL thành các lệnh gọi AWS API tương ứng để tạo lập hạ tầng mạng, máy chủ và cân bằng tải.

2. TLS & Local Providers: Tạo khóa bảo mật RSA-4096 hoàn toàn bằng phần mềm và ghi file private key .pem cục bộ xuống ổ đĩa.

3. Kubernetes Provider: Kết nối qua cổng proxy 8081 trên IP công cộng của EC2 để triển khai các Pod ứng dụng.

Mọi trạng thái tài nguyên đều lưu tại file terraform.tfstate giúp Terraform so sánh và cập nhật gia tăng (incremental changes) một cách tối ưu.

Dựng môi trường VPC tách biệt trên cloud. Thiết lập Internet Gateway, bảng định tuyến công cộng và cấu hình 2 Security Groups (ec2-sg và alb-sg) để cô lập mạng.

Files chạy: vpc.tf, security_groups.tf Tài nguyên: aws_vpc.main, aws_security_group.ec2-sg, aws_security_group.alb-sg
4

Giai đoạn 4: Tạo EC2 & Cân bằng tải ALB

EC2 Instance được tạo và nhận key SSH cùng kịch bản cài đặt tự động. ALB cũng được dựng đồng thời cùng Target Group trỏ vào port 30080 trên EC2.

Files chạy: ec2.tf, alb.tf Tài nguyên: aws_instance.minikube, aws_lb.app, aws_lb_target_group.app
5

Giai đoạn 5: Khởi chạy Bootstrap script (user_data.sh)

Trên máy ảo EC2, kịch bản scripts/user_data.sh bắt đầu chạy ngầm: Cập nhật hệ thống ➔ Cài đặt Docker ➔ Cài đặt kubectl & Kind ➔ Dựng cluster Kind Kubernetes ➔ Chạy ngầm kubectl proxy tại port 8081.

Tập lệnh chạy: scripts/user_data.sh (trên EC2) Logs kiểm tra: /var/log/user-data.log (truy cập bằng SSH để xem)
6

Giai đoạn 6: Đợi Proxy Kubernetes sẵn sàng (Wait state)

Trong lúc EC2 chạy script (mất khoảng 2-3 phút), tài nguyên null_resource.wait_for_minikube chạy vòng lặp tại máy local để kiểm tra tình trạng kết nối tới proxy port 8081. Khi proxy phản hồi HTTP 200 OK, quá trình đợi kết thúc.

Files chạy: ec2.tf Tài nguyên: null_resource.wait_for_minikube (Local exec)
7

Giai đoạn 7: Kubernetes Provider triển khai ứng dụng

Kubernetes Provider kết nối qua endpoint http://<EC2-IP>:8081. Nó bắt đầu tạo Namespace lab-cd9, ConfigMap chứa trang index.html, Deployment chạy Nginx Pod, và Service NodePort 30080.

Files chạy: kubernetes.tf Tài nguyên: kubernetes_namespace.web, kubernetes_config_map.web_html, kubernetes_deployment.web, kubernetes_service.web
8

Giai đoạn 8: Hoàn thành & Xuất Output

Hệ thống được khởi tạo thành công. Terraform lưu trạng thái cuối cùng vào file state và in ra các outputs quan trọng như DNS của ALB, IP công cộng của EC2 và lệnh SSH.

Files chạy: outputs.tf Đầu ra: alb_dns_name, instance_public_ip, ssh_command

So Sánh Công Nghệ

Tại Sao Sử Dụng Kind Thay Vì Minikube?

Kind (Được Lựa Chọn)
  • Không yêu cầu ảo hóa lồng
    Chạy trực tiếp dưới dạng các container Docker, hoàn hảo cho EC2 t3.medium không hỗ trợ Nested Virtualization.
  • Expose Port tự nhiên
    Ánh xạ cổng từ container node ra host EC2 trực tiếp, giúp Application Load Balancer trỏ thẳng vào target group qua port 30080 dễ dàng.
  • Khởi động rất nhanh & Nhẹ
    Không cần boot một hệ điều hành ảo hóa (Hypervisor) toàn phần, chỉ cần Docker Engine sẵn có giúp giảm thời gian bootstrap đáng kể.
Minikube (Không Lựa Chọn)
  • Yêu cầu ảo hóa lồng (Nested Virtualization)
    Các driver mặc định (KVM2, VirtualBox) đòi hỏi phần cứng hỗ trợ ảo hóa trong VM. AWS EC2 dòng phổ thông không hỗ trợ tính năng này.
  • Driver Docker hạn chế kết nối
    Nếu dùng driver docker của Minikube, NodePort bị cô lập trong mạng nội bộ docker bridge của minikube host, Load Balancer không kết nối tới được.
  • Khởi động nặng nề
    Minikube nặng hơn do mang theo nhiều tính năng quản lý local GUI/Addons không cần thiết cho máy chủ headless.
Kết luận: Sử dụng Kind là giải pháp tối ưu nhất để tự động hóa hoàn toàn hạ tầng Kubernetes trên môi trường Cloud EC2 Headless. Nó cung cấp sự cân bằng tuyệt vời giữa hiệu năng khởi chạy, khả năng tương thích phần cứng đám mây và tính ổn định khi kết nối với nhà cung cấp Terraform.
